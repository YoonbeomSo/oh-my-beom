# CLAUDE.md

oh-my-beom 플러그인의 최상위 지침.

한국어로 응답하세요. 코드와 커밋 메시지도 한국어를 기본으로 합니다.

---

## 작업 범위

- **PR 생성까지만.** `gh pr merge` 등 머지 명령은 절대 실행하지 마세요. 사용자가 직접 머지를 요청하더라도 거절하고, PR 링크를 제공하여 직접 머지하도록 안내하세요.
- 보호 브랜치(`main`, `master`, `develop`, `test`. `dev`)에서 직접 커밋하지 마세요. 작업 브랜치를 먼저 생성하세요.
- 커밋 메시지에 `Co-Authored-By` 트레일러를 추가하지 마세요.

## 행동 원칙 요약

상세 규칙은 `rules/` 디렉토리를 참조하세요.

1. **코딩 전에 생각하라** - 추측하지 마라. 읽기 우선. 환각 금지.
2. **단순함 우선** - 최소한의 코드. 오버엔지니어링 금지.
3. **외과적 변경** - 건드려야 할 것만 건드려라.
4. **목표 중심 실행** - 성공 기준을 정의하고 검증하라.
5. **점진적 실행** - 작은 단위로 나눠서 검증하라.
6. **실패 대응** - 근본 원인부터 파악하라.
7. **피드백 반영** - 지적받은 실수를 반복하지 마라.

## 기본 워크플로우

```
/lens (정책가) → /research (조사관) → /plan (계획자) → /dev (개발자)
```

| 순서 | 스킬 | 별칭 | 역할 | 산출물 | 후속 스킬이 참조 |
|------|------|------|------|--------|----------------|
| 1 | `/lens` | 정책가 | 비즈니스 정책/영향도 분석 | `.dev/lens-report.md` | /plan, /dev |
| 2 | `/research` | 조사관 | 코드베이스 탐색/기술 조사 | `.dev/research-report.md` | /plan, /dev |
| 3 | `/plan` | 계획자 | 계획서 수립 (TODO + 완료 기준) | `docs/plan/*.md` | /dev |
| 4 | `/dev` | 개발자 | PRD → 설계 → 구현 → 리뷰 → PR | 코드 + PR | - |

> 각 스킬은 산출물을 `.dev/`에 저장하고, 후속 스킬이 자동으로 감지하여 참조합니다. 독립 실행도 가능합니다.

### 상황별 시작점

| 상황 | 시작 | 예시 |
|------|------|------|
| 간단한 버그 수정 | `/dev` | `/dev [JIRA-123] 재고 차감 버그 수정` |
| 긴급 핫픽스 | `/dev --hotfix` | `/dev --hotfix [JIRA-999] 결제 오류 긴급` |
| 코드 파악 필요 | `/research` → `/dev` | `인증 모듈 구조 분석해` → `/dev` |
| 큰 기능 개발 | `/research` → `/plan` → `/dev` | 조사 → 계획 → 개발 |
| 정책이 복잡 | `/lens` → `/plan` → `/dev` | 정책 분석 → 계획 → 개발 |
| 풀코스 | `/lens` → `/research` → `/plan` → `/dev` | 정책 → 조사 → 계획 → 개발 |

> `/persist`는 순서 무관. 어떤 작업이든 "끝까지 멈추지 말고 해"라고 할 때 사용.

## 에이전트 역할 경계

| 역할 | 할 수 있는 것 | 할 수 없는 것 |
|------|-------------|-------------|
| Explorer | 파일 읽기, 구조 파악, 패턴 수집 | 파일 수정, 코드 작성 |
| Planner | 계획 수립, 질문, 선택지 제시 | 코드 구현 |
| Coder | 코드 작성, 수정, 테스트 | PR 머지, 보호 브랜치 커밋 |
| Critic | 리뷰, 평가, 판정 | 직접 코드 수정 |
