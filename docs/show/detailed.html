Stagnation Detection — 정체 감지 시 에스컬레이션

/dev의 Phase 4(구현) 중 에이전트가 진전 없이 반복하면 정체 패턴을 감지하고, 전문 에이전트로 에스컬레이션합니다.
SPINNING
같은 에러가 2회 반복
→ 1차: hacker (제약 우회) → 2차: researcher (근본 원인)
OSCILLATION
접근법 A ↔ B 반복 전환
→ 1차: architect (설계 재검토) → 2차: 사용자 결정
NO_DRIFT
반복해도 코드 변경 없음
→ 1차: hacker (제약 분석) → 2차: researcher (탐색)
DIMINISHING_RETURNS
스코프 축소, 개선 없음
→ 1차: simplifier (스코프 축소) → 2차: 사용자 방향 결정

5Hybrid Team Agent Model

에이전트 실행 방식이 두 가지로 분리됩니다. 핵심 에이전트(coder, qa-manager, security-auditor, design-critic)는 Agent(teammate)로 실행되어 병렬 협업이 가능하고, 나머지는 Task(subagent)로 실행됩니다. config.json의 teamAgent 섹션에서 활성화/모드/단계를 설정합니다.
Agent (teammate)
  • coder (opus) — 코드 수정
  • qa-manager (opus) — 리뷰
  • security-auditor (opus) — 보안 감사
  • design-critic (opus) — 설계 비판

병렬 실행 가능, idle 감지, 자동 재개

Task (subagent)
  • product-owner (sonnet) — 요구사항
  • architect (opus) — 설계
  • researcher (sonnet) — 리서치
  • hacker (sonnet) — 정체 돌파
  • simplifier (sonnet) — 스코프 축소

순차 실행, 호출-응답 방식

6Verification Loop — Plan-driven Development

Plan 파일은 docs/plan/ 디렉토리에 machine-parseable TODO 포맷으로 저장됩니다. 각 TODO 항목에 completion criteria 태그가 붙어 있어 자동 검증이 가능합니다.

Plan 파일 포맷

# docs/plan/login-feature.md
- [ ] AuthController 구현 [FILE_EXISTS:src/main/java/.../AuthController.java]
- [ ] JWT 토큰 발급 로직 [TEST:AuthServiceTest]
- [ ] 린트 통과 [LINT]
- [ ] 비밀번호 암호화 [CRITERIA:BCrypt 사용]

Verification 메커니즘

coder 작업 완료 TaskCompleted → task-verifier plan criteria check → todo-verifier TODO 상태 확인 → plan-visualizer 진행률 시각화 → all criteria met → Review 진행 not met → coder에게 되돌림
Agent(teammate) idle → idle-checker (TeammateIdle) → 남은 TODO 확인 → auto-continue next TODO

Mechanical Checks (config.json → verification)

Tag검증 방법예시
[LINT]프로젝트 린트 명령 실행eslint, ktlint 등 통과 여부
[TEST]지정된 테스트 실행[TEST:AuthServiceTest] → 해당 테스트 통과 확인
[FILE_EXISTS]파일 존재 여부 확인[FILE_EXISTS:src/.../AuthController.java]
[CRITERIA]사람이 읽을 수 있는 조건[CRITERIA:BCrypt 사용] → 에이전트가 코드에서 확인

8/plan — 4-Mode 계획 전략

사용자 입력을 분석하여 4가지 계획 모드 중 하나를 자동 선택합니다. 간단한 작업은 Direct, 모호한 요구는 Interview, 중요한 결정은 Consensus, 기존 계획 검증은 Review.
Direct
조건: "간단히", "빠르게", 또는 구체적 지시
Explorer (haiku) → 계획 직접 작성

코드베이스를 빠르게 탐색한 후 바로 계획을 작성합니다. 가장 빠른 모드.

Interview
조건: "인터뷰", "질문", 또는 모호한 요구사항
질문 5개 → AskUser → 분석 → 계획

Q&A를 통해 요구사항을 명확히 한 후 계획을 수립합니다. product-owner의 요구사항 분석 7단계와 유사.

Consensus
조건: "합의", "검증", "여러 관점", 또는 중대한 결정
Plan Agent x3 → Critic → APPROVED / REVISE / REJECT

3개의 Plan agent가 각각 다른 관점(단순성 / 성능 / 유지보수성)으로 계획을 작성 → Critic이 합의안 도출. design-critic 패턴과 동일한 APPROVED/REVISE/REJECT 게이트.

Review
조건: 기존 계획 언급, "리뷰", "검토"
기존 계획 → Critic → 피드백 루프

기존 계획을 Critic이 분석하여 개선점을 제안합니다. REVISE/REJECT 시 반복.

9/research — 3-Phase 리서치 파이프라인

코드베이스를 체계적으로 조사하여 구조화된 보고서를 작성합니다. 입력 유형(architecture / performance / bug / general)에 따라 조사 초점이 달라집니다.
사용자 질문 → Explorer (haiku) 프로젝트 스캔
파일/모듈 식별 → Researcher (sonnet) 코드 패턴 수집
문서 분석 → Analyzer (opus) 결과 병합
구조화된 보고서 → 보고서
출력 형식: 현황 → 문제점 → 개선 제안 → 실행 방안

11Config + Rules ↔ 컴포넌트 참조 맵

config/config.json과 rules/의 규칙 파일들은 여러 컴포넌트에서 공유 참조됩니다.
설정/규칙참조하는 컴포넌트용도
config.json → issueKeyPattern commit skill, pull-request skill 브랜치명에서 이슈키 추출 시 정규식 패턴 (([A-Z]+-\d+))

config.json → contextLimits dev skill (context slicing) 에이전트별 토큰 예산. product-owner(8K), coder(16K), qa-manager(12K) 등으로 불필요한 컨텍스트 전달 방지

rules/behavior.md 모든 에이전트 (session-start로 주입) 행동 원칙 §1~§7. "코딩 전에 생각하라", "단순함 우선" 등 모든 에이전트의 기본 행동 원칙
rules/git-workflow.md pre-tool-guard hook, commit skill, coder agent 브랜치 전략 (feature only), 한국어 커밋 메시지, Co-Authored-By 금지
rules/workspace-structure.md worktree skill, dev skill (setup phase) 디렉토리 구조 규칙. worktree 생성 위치, 환경 파일 복사 규칙
