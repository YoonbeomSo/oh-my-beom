0
세션 컨텍스트 (이미 주입됨) SessionStart
세션 시작 시 session-start hook이 이미 실행되어 활성 계획(IN_PROGRESS)이 있으면 진행 상황이 컨텍스트에 주입됨. 행동 원칙은 루트 CLAUDE.md와 rules/에서 자동 로딩됨.
1
prompt-router 키워드 감지 UserPromptSubmit Hook
사용자 메시지가 Claude에 도달하기 전에 prompt-router hook이 먼저 실행됩니다.
stdin (JSON) {"prompt": "로그인 기능 개발해줘"}
1.python3 -c로 prompt 필드 추출
2.grep -qi "개발해\|dev \|구현해\|만들어줘" → "개발" 매칭!
3.매칭된 스킬: oh-my-beom:dev
stdout (JSON) {"hookSpecificOutput": {"additionalContext": "[KEYWORD DETECTED: dev] You MUST invoke Skill: oh-my-beom:dev ..."}}
2
Claude가 스킬 호출 지시를 수신 Context Injection
Claude는 사용자 메시지 + additionalContext를 함께 받습니다. Context에 "You MUST invoke Skill: oh-my-beom:dev"라는 지시가 있으므로 Claude는 자동으로 Skill tool을 호출합니다.
3
/dev 스킬 활성화 Skill Tool
/dev SKILL.md (680줄)의 전체 워크플로우가 Claude의 지시사항으로 로드됩니다. Claude는 이제 6-Phase 파이프라인을 순서대로 실행합니다.
── Phase 1: Setup ──
4
워크트리 생성 + 코드맵 초기화 Setup Phase
1.config/config.json에서 프로젝트 타입 감지 (build.gradle.kts → kotlin-java)
2./worktree create login-feature 실행 → ../worktrees-{project}/login-feature 생성
3..dev/codemap.md 초기화 — 관련 파일 목록 수집 시작
── Phase 2: Requirements ──
5
product-owner 에이전트 Agent
product-owner (sonnet)
PRD(Product Requirements Document)를 작성합니다.
1.문제 재해석: "로그인 기능 추가" → "인증되지 않은 사용자의 서비스 접근을 제어하는 문제" (사용자/비즈니스/시스템 관점)
2.애매함 도출: "JWT vs 세션?", "소셜 로그인 포함?", "실패 횟수 제한?" 등 정책/경계/확장 질문
3.Q&A 루프: 사용자에게 선택지 + 영향도를 함께 제시하여 확인
4.PRD 작성: Acceptance Criteria 포함
출력 artifact .dev/prd.md → 다음 Phase의 architect에게 전달
── Phase 3: Design ──
6
architect 에이전트 Agent
architect (opus)
PRD를 기반으로 기술 설계를 작성합니다.
1..dev/prd.md + .dev/codemap.md 읽기
2.API 설계: POST /api/v1/auth/login, Request/Response DTO, 에러 코드 (내장된 api-design 원칙 적용)
3.엔티티 설계: User 엔티티, 연관관계, @ManyToOne LAZY (내장된 jpa-entity-design 원칙 적용)
4.구현 단계를 순서대로 나열
출력 artifact .dev/design.md → design-critic에게 전달
7
design-critic 에이전트 Review Loop
design-critic (opus)
설계의 가정을 도전합니다.
1.CHALLENGE: "JWT 토큰 탈취 시 세션 무효화 전략이 없음"
2.SIMPLIFY: "Spring Security 전체 도입보다 필터 하나로 충분"
3.ROOT-CAUSE: "인증이 필요한 진짜 이유가 API 보호인지 사용자 식별인지 구분 필요"
판정 APPROVED → Phase 4로 진행 | REVISE → architect가 수정 후 재제출 | REJECT → 설계 재시작
── Phase 4: Implement ──
8
coder 에이전트 Agent (유일한 코드 수정자)
coder (opus) — Agent(teammate)
확정된 설계에 따라 코드를 구현합니다.
1..dev/design.md + .dev/codemap.md 기반으로 구현
2.Spring Boot 컨벤션 적용: AuthReadService (조회) + AuthService (인증)
3.DTO/Entity 분리, @Transactional(readOnly=true), LAZY fetch
4.자체 검토: .dev/self-check.md 작성
⚡
pre-tool-guard 브랜치 보호 PreToolUse Hook
coder가 git commit을 시도할 때마다 이 hook이 개입합니다.
체크 현재 브랜치 = login-feature (feature branch) → ✅ ALLOW
만약 main 브랜치였다면 ❌ DENY — "Protected branch: main 브랜치에서는 직접 커밋할 수 없습니다"
── Verification Loop ──
V
Verification Loop TaskCompleted + TeammateIdle Hooks
Plan-driven Development: docs/plan/에 저장된 계획의 TODO 기준으로 완료를 검증합니다.
1.TaskCompleted hook (task-verifier): 작업 완료 시 plan의 completion criteria 체크 ([LINT], [TEST], [FILE_EXISTS], [CRITERIA]). 미충족 시 block
2.TeammateIdle hook (idle-checker): teammate가 idle 상태일 때 남은 TODO 확인 → 자동 재개
3.todo-verifier agent: plan 파일의 machine-parseable TODO 포맷 검증
4.plan-visualizer agent: 진행 상황 시각화 → docs/show/에 HTML 생성
판정 모든 criteria 충족 → Review Phase 진행 | 미충족 → Implement로 되돌림
── Phase 5: Review ──
9
qa-manager + security-auditor (병렬) Review Loop
qa-manager (opus) — Agent(teammate) security-auditor (opus) — Agent(teammate)
두 에이전트가 병렬로 리뷰합니다. 둘 다 읽기 전용.
QA45조항 체크리스트 적용: Service Read/Write 분리 ✅, 삼항 연산자 ✅, indent 2단계 ✅, null 반환 ❌ ...
SECZero Trust 감사: SQL Injection ✅, 비밀번호 평문 저장 ❌, CSRF 보호 ✅ ...
결과 CRITICAL 이슈 발견 → coder에게 수정 요청 → 수정 후 재리뷰 (max 2회 반복)
출력 artifact .dev/trust-ledger.md — 보안 감사 결과 기록
── Phase 6: Complete ──
10
/commit 스킬 호출 Skill
