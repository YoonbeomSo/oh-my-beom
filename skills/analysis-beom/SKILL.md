분석 오케스트레이터. 코드베이스 또는 비즈니스 정책을 분석하고 보고서를 생성한다.

항상 한국어로 응답한다.

## 절대 원칙

1. **읽기 전용.** 소스 코드를 수정하지 않는다.
2. **보고서를 반드시 생성한다.** 분석 결과를 `docs/result/result_{분석주제}.md`에 저장한다.

## 인자

- `ARGS`: 분석 대상 설명. Jira URL이 포함될 수 있다.

ARGS 없이 호출 시: "분석할 대상을 설명해주세요. 예: `/analysis-beom 결제 모듈 아키텍처 분석`"

---

# 실행 플로우

## Phase 1: 분석 유형 감지

ARGS에서 키워드로 분석 유형을 결정한다:

| 키워드 | 유형 | 설명 |
|--------|------|------|
| 정책, 비즈니스, 영향도, 규칙 | **정책 분석** | 코드에서 비즈니스 정책을 탐지하고 영향도 분석 |
| 그 외 | **코드 분석** | 아키텍처, 성능, 버그 원인, 구조 분석 |

Jira URL/이슈키가 있으면 `Skill("oh-my-beom:fetch-jira-issue")` 호출.

## Phase 2: 탐색 (Explorer)

Explore 에이전트를 실행하여 프로젝트를 스캔한다:

```
Agent(subagent_type="Explore", prompt="""
{분석 유형}을 위해 프로젝트를 탐색해주세요.

분석 대상: {ARGS}
Jira 컨텍스트: {있으면}

탐색 항목:
- 디렉토리 구조
- 관련 파일 및 모듈
- 핵심 패턴 및 의존성
- {정책 분석이면: 비즈니스 규칙이 구현된 위치}
- {코드 분석이면: 아키텍처 패턴, 성능 관련 코드}
""")
```

## Phase 3: 심층 조사 (Researcher)

탐색 결과를 기반으로 심층 조사를 수행한다:

```
Agent(subagent_type="Explore", prompt="""
Phase 2 탐색 결과를 기반으로 심층 조사를 수행해주세요.

탐색 결과: {Phase 2 결과}

조사 항목:
- 핵심 파일 상세 분석
- 의존성 추적
- {정책 분석이면: 정책 변경 시 영향 범위, 관련 테스트}
- {코드 분석이면: 문제점, 개선 기회, 기술 부채}
""")
```

## Phase 4: 보고서 작성

분석 결과를 종합하여 보고서를 생성한다:

`docs/result/result_{분석주제}.md`:

```markdown
# 분석 보고서: {분석 주제}

- 유형: 코드 분석 / 정책 분석
- 대상: {분석 대상}
- 생성일: {date}

## 요약
{핵심 발견 1-3줄}

## 분석 결과

### {분석 항목 1}
{상세 내용}

### {분석 항목 2}
{상세 내용}

## 권장 조치
{있으면}

## 다음 단계
- 개발이 필요하면: `/dev-beom {작업 설명}`
- 버그 수정이면: `/fix-beom {버그 설명}`
```

마지막에 사용자에게 후속 작업 안내를 출력한다.
