# Plan Skill - 4-Mode 계획 수립

한국어로 응답한다.

## 모드 감지 (Decision Tree)

사용자 입력을 분석하여 아래 기준으로 모드를 결정한다:

```
사용자 입력 분석
├── 기존 계획 파일 경로가 언급됨?
│   └── YES -> [Review 모드]
├── "합의", "여러 관점", "비교", "consensus" 포함?
│   └── YES -> [Consensus 모드]
├── 요구사항이 구체적이고 명확한가? (기술 스택, 범위, 목표가 특정됨)
│   └── YES -> [Direct 모드]
└── 그 외 (애매하거나 탐색적)
    └── [Interview 모드]
```

모드를 결정하면 사용자에게 감지된 모드를 알리고 진행한다:
```
[Plan] {모드명} 모드로 진행합니다.
```

## 이전 스킬 산출물 참조

계획 수립 전에 이전 스킬의 산출물을 확인한다:

1. `${PROJECT_ROOT}/.dev/lens-report.md`가 있으면 Read하여 정책 분석 결과를 계획에 반영한다.
2. `${PROJECT_ROOT}/.dev/research-report.md`가 있으면 Read하여 조사 결과를 계획에 반영한다.
3. 두 파일 모두 없으면 건너뛴다 (독립 실행).

참조된 내용은 계획서의 "배경" 섹션에 반영하고, 정책 제약은 "리스크" 섹션에, 기술 조사 결과는 "기술 결정"에 반영한다.

## Direct 모드

요구사항이 명확할 때. 탐색 -> 계획 작성을 바로 수행한다.

### 절차

1. **탐색**: Explore 에이전트(`Agent` tool, `type: explore`)로 프로젝트 구조와 관련 코드를 스캔한다.
   - 프로젝트 루트의 주요 파일/디렉토리 구조
   - 요구사항과 관련된 기존 코드 패턴
   - 의존성 및 기술 스택 확인

2. **계획 작성**: 탐색 결과를 바탕으로 계획서를 작성한다.

### 계획서 형식

```markdown
# {이슈키}: {계획 제목}

## 메타데이터
- 상태: IN_PROGRESS
- 생성일: {date}
- 브랜치: (미정)
- 베이스: (미정)

## 배경
- 현재 상황 요약
- 해결하려는 문제

## 목표
- 성공 기준 (검증 가능한 형태)

## 기술 결정
| 결정 사항 | 선택 | 근거 |
|-----------|------|------|

## 최종 완료 기준
- [ ] [LINT] 린트 통과
- [ ] [TEST] 테스트 전체 통과
- [ ] [BUILD] 빌드 성공
- [ ] [CRITERIA] {목표에서 도출한 검증 가능 조건}

## 실행 단계

### Step 1: {단계 제목}
상태: TODO

#### TODO 항목
- [ ] [TODO-1.1] {구현 항목}
  - 완료 기준: [FILE_EXISTS] {경로} | [TEST_PASSES] {패턴} | [GREP_MATCH] {패턴} in {파일} | [CRITERIA] {설명}
  - 검증 상태: PENDING
  - 비고:

#### Step 완료 기준
- [ ] [STEP-CRITERIA] {단계 검증 조건}

## 리스크
- [리스크] → 대응: [완화 방안]

## 범위 외
- 이번 계획에 포함하지 않는 것

## 변경 이력
- {date} 초기 계획 수립
```

### 완료 기준 자동 태깅 규칙

TODO 항목의 완료 기준을 작성할 때, 기계적으로 검증 가능한 기준은 자동으로 적절한 태그를 부여한다:

| 작업 유형 | 태그 | 예시 |
|-----------|------|------|
| 파일 생성 | `[FILE_EXISTS]` | `[FILE_EXISTS] src/utils/parser.ts` |
| 테스트 작성 | `[TEST_PASSES]` | `[TEST_PASSES] parser.test.ts` |
| 패턴 구현 | `[GREP_MATCH]` | `[GREP_MATCH] "export class Parser" in src/utils/parser.ts` |
| 코드 분석 필요 | `[CRITERIA]` | `[CRITERIA] API 응답 시간 200ms 이하` |

- 파일 생성이 목적인 항목은 반드시 `[FILE_EXISTS]`를 사용한다.
- 테스트 파일 작성 또는 테스트 통과가 목적인 항목은 `[TEST_PASSES]`를 사용한다.
- 특정 코드 패턴이 존재해야 하는 항목은 `[GREP_MATCH]`를 사용한다.
- `[CRITERIA]`는 위 태그로 표현할 수 없는, 코드 분석이 필요한 기준에만 사용한다.

## Interview 모드

요구사항이 애매하거나 탐색적일 때. 질문을 통해 명확화한 뒤 계획을 수립한다.

### 절차

1. **질문 생성**: 사용자 입력을 분석하여 5개 핵심 질문을 생성한다.
   - 질문 유형: 목표 확인, 범위 확인, 기술 제약, 우선순위, 성공 기준
   - 각 질문에 선택지 + 영향도를 함께 제시한다

2. **질문 제시**: 5개 질문을 한 번에 제시한다.
   ```
   계획 수립을 위해 몇 가지 확인이 필요합니다:

   1. [질문]
      - A: [선택지] -> [영향]
      - B: [선택지] -> [영향]

   2. [질문]
   ...
   ```

3. **답변 분석**: 사용자 답변을 종합 분석한다.

4. **계획 작성**: Direct 모드와 동일한 형식으로 계획서를 작성한다.

## Consensus 모드

복잡한 결정이 필요할 때. 3개 관점에서 계획을 수립하고 병합한다.

### 절차

1. **3개 관점 계획 수립**: 각각 Agent tool을 사용하여 병렬로 실행한다.

   | 관점 | 역할 | 초점 |
   |------|------|------|
   | Pragmatist | 실용주의자 | 최소 비용, 빠른 구현, 기존 인프라 활용 |
   | Architect | 설계자 | 확장성, 유지보수성, 기술 부채 최소화 |
   | User-Advocate | 사용자 대변인 | 사용자 경험, 접근성, 실제 사용 시나리오 |

   각 에이전트에게 동일한 요구사항을 전달하되, 관점을 명시한다:
   ```
   당신은 {관점}입니다. 다음 요구사항에 대해 {초점}을 중심으로 계획을 수립하세요.
   요구사항: {사용자 입력}
   ```

2. **병합**: 3개 계획의 공통점과 차이점을 분석하여 통합 계획을 작성한다.

3. **Critic 리뷰**: 통합 계획에 대해 비평 에이전트를 실행한다.
   - 평가 기준: 실현 가능성, 일관성, 누락된 고려사항
   - 판정: **APPROVED** / **REVISE** / **REJECT**

4. **결과 처리**:
   - APPROVED: 최종 계획서로 사용자에게 제시
   - REVISE: 지적 사항 반영하여 계획 수정 후 재리뷰
   - REJECT: 근본적 문제점을 사용자에게 공유하고 방향 재논의

### 출력 형식

```markdown
# {이슈키}: {계획 제목}

## 관점별 분석 요약

### Pragmatist
- 핵심 제안: ...
- 장점: ... | 단점: ...

### Architect
- 핵심 제안: ...
- 장점: ... | 단점: ...

### User-Advocate
- 핵심 제안: ...
- 장점: ... | 단점: ...

## 통합 계획
{위 "계획서 형식"의 메타데이터~변경 이력 섹션과 동일한 형식으로 작성}

## Critic 리뷰
- 판정: {APPROVED/REVISE/REJECT}
- 평가: ...
- 보완 사항: ...
```

## Review 모드

기존 계획을 검토하고 개선한다.

### 절차

1. **계획 로드**: 사용자가 지정한 파일 또는 이전 세션의 계획을 읽는다.

2. **Critic 분석**: 기존 계획에 대해 다음을 평가한다.
   - 실현 가능성: 기술적으로 구현 가능한가?
   - 완전성: 누락된 단계나 고려사항이 있는가?
   - 리스크: 식별되지 않은 위험이 있는가?
   - 효율성: 불필요한 단계가 있는가?

3. **개선안 제시**: 문제점과 개선안을 구조화하여 제시한다.
   ```
   ## 리뷰 결과

   ### 강점
   - ...

   ### 개선 필요
   | 항목 | 현재 | 제안 | 이유 |
   |------|------|------|------|

   ### 누락 사항
   - ...
   ```

4. **반복**: 사용자 피드백을 반영하여 계획을 수정한다.

## 행동 규칙

- 모드를 사용자에게 알리고, 모드가 부적절하다고 판단되면 다른 모드를 제안하라.
- 계획서에는 구현 코드를 포함하지 마라. 코드 수준 상세는 `/dev` 스킬에서 처리한다.
- Agent tool 사용 시 `type: explore`로 읽기 전용 탐색을 수행하라. 파일 수정은 하지 마라.

### 계획 파일 저장

모든 모드(Direct, Interview, Consensus, Review)에서 계획 작성이 완료되면 반드시 파일로 저장한다.

1. **저장 경로**: `docs/plan/{파일명}-plan.md`
   - 디렉토리가 없으면 `mkdir -p docs/plan`으로 생성한다 (Bash tool 사용).

2. **파일명 결정**:
   - 사용자 입력에서 이슈 키(`config/config.json`의 `issueKey.pattern`에 매칭되는 패턴, 예: `JIRA-123`)가 발견되면 이슈 키를 소문자로 변환하여 사용한다. (예: `jira-123-plan.md`)
   - 이슈 키가 없으면 계획 제목에서 핵심 키워드를 추출하여 slug를 생성한다. (예: `api-인증-개선-plan.md`)

3. **저장 후 처리**:
   - `config/config.json`의 `verification.autoVisualize`가 `true`이면, 저장 완료 후 plan-visualizer 에이전트를 호출하여 시각화를 수행한다.

4. **Review 모드**: 기존 계획 파일을 수정한 경우, 동일한 경로에 덮어쓴다. 변경 이력 섹션에 수정 내용을 추가한다.

### 다음 액션 제안

- 계획 작성 후 사용자에게 다음 액션을 제안하라: `/dev`로 구현 시작, 계획 수정, 또는 추가 조사.
