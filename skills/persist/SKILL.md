# Persist Skill - 끝까지 실행 모드

한국어로 응답한다.

## 개요

작업을 중단 없이 완료까지 수행하는 모드. 6단계 파이프라인으로 진행하며, Phase 3~5를 모든 이슈가 해결될 때까지 반복한다.

```
[Phase 1: 작업 분석] -> [Phase 2: 실행] -> [Phase 3: 리뷰] -> [Phase 4: 검증] -> [Phase 5: 보완] -> [Phase 6: 완료]
                                              ^                                        |
                                              |________________________________________|
                                                    이슈 발견 시 루프
```

---

## Phase 1: 작업 분석

사용자 요청을 파싱하여 실행 계획을 수립한다.

### 수행 사항

1. **요구사항 파싱**
   - 사용자 요청에서 작업 항목을 추출한다
   - 암묵적 요구사항을 식별한다 (예: "API 추가" -> 테스트도 필요)

2. **성공 기준 정의**
   - 각 작업 항목의 완료 조건을 검증 가능한 형태로 정의한다
   - 예: "빌드 성공", "테스트 통과", "API 응답 확인"

3. **실행 순서 결정**
   - 의존 관계를 분석하여 실행 순서를 결정한다
   - 병렬 실행 가능한 항목을 식별한다

### 출력

```
[Persist Phase 1/6: 작업 분석 완료]

작업 항목:
1. {항목} -> 성공 기준: {기준}
2. {항목} -> 성공 기준: {기준}
...

병렬 가능: {항목 번호 그룹}
예상 단계: {N}단계
```

---

## Phase 2: 실행

계획에 따라 구현을 수행한다.

### 수행 사항

1. **단계별 구현**
   - Phase 1에서 정한 순서대로 구현한다
   - 각 단계 완료 시 상태를 보고한다

2. **병렬 실행**
   - 독립적인 작업은 Agent tool로 병렬 수행한다
   - 에이전트 간 공유 상태가 필요한 작업은 순차 실행한다

3. **중간 보고**
   - 각 작업 항목 완료 시:
     ```
     [Persist Phase 2/6: 실행 중] {완료}/{전체} 항목 완료
     - [완료] {항목명}
     - [진행 중] {항목명}
     ```

---

## Phase 3: 리뷰

구현된 코드를 자가 점검한다.

### 점검 항목

1. **코딩 컨벤션 준수**
   - 프로젝트의 기존 스타일과 일치하는가?
   - 네이밍 규칙을 따르는가?

2. **행동 원칙 준수** (`rules/behavior.md` 참조)
   - 불필요한 변경이 포함되어 있지 않은가? (외과적 변경)
   - 오버엔지니어링이 아닌가? (단순함 우선)
   - 요청 범위를 벗어난 변경이 없는가?

3. **설계 품질**
   - 책임이 명확하게 분리되어 있는가?
   - 에러 처리가 적절한가?
   - 엣지 케이스를 고려했는가?

### 출력

```
[Persist Phase 3/6: 리뷰 완료]

이슈: {N}건
- [{심각도}] {이슈 설명} @ {파일:라인}
...

이슈 없음이면: "리뷰 통과"
```

---

## Phase 4: 검증

빌드와 테스트를 실행하여 기능을 검증한다.

### 수행 사항

1. **프로젝트 타입 감지**
   - `config/config.json`의 `projectTypes`를 참조하여 빌드/테스트 명령을 결정한다

2. **빌드 실행**
   - 빌드 명령을 실행하고 결과를 확인한다
   - 타임아웃: `config/config.json`의 `timeouts.build` 참조

3. **테스트 실행**
   - 테스트 명령을 실행하고 결과를 확인한다
   - 타임아웃: `config/config.json`의 `timeouts.test` 참조

4. **린트 실행** (해당 시)
   - 린트 명령을 실행하고 결과를 확인한다

### 출력

```
[Persist Phase 4/6: 검증 완료]

빌드: {성공/실패}
테스트: {성공/실패} ({통과}/{전체})
린트: {성공/실패}

실패 항목:
- {실패 내용}
...

모두 성공이면: "검증 통과"
```

---

## Phase 5: 보완

Phase 3 (리뷰)과 Phase 4 (검증)에서 발견된 이슈를 수정한다.

### 수행 사항

1. **이슈 목록 통합**
   - Phase 3의 리뷰 이슈 + Phase 4의 검증 실패를 통합한다

2. **우선순위 결정**
   - 빌드 실패 > 테스트 실패 > 린트 오류 > 리뷰 이슈

3. **수정 실행**
   - 각 이슈를 순서대로 수정한다
   - 수정 후 관련 검증을 재실행한다

4. **루프 판단**
   - 모든 이슈가 해결되었는가?
   - YES -> Phase 6으로 진행
   - NO -> Phase 3으로 돌아가 재리뷰

### 루프 제한

- 최대 3회 루프. 3회 후에도 미해결 이슈가 있으면 사용자에게 보고하고 판단을 요청한다.

### 출력

```
[Persist Phase 5/6: 보완 완료]

수정된 이슈: {N}건
잔여 이슈: {N}건

-> Phase 3으로 돌아갑니다 (루프 {M}/3)
또는
-> 모든 이슈 해결. Phase 6으로 진행합니다.
```

---

## Phase 6: 완료

최종 확인 및 결과 보고.

### 수행 사항

1. **최종 검증**
   - 빌드 + 테스트 + 린트 전체 재실행
   - 성공 기준 달성 여부 확인

2. **결과 보고**

```
[Persist 완료]

## 작업 결과

| # | 항목 | 상태 | 비고 |
|---|------|------|------|
| 1 | {항목} | 완료 | {상세} |
...

## 변경 파일
- {파일 경로}: {변경 요약}
...

## 검증 결과
- 빌드: 성공
- 테스트: {통과}/{전체} 통과
- 린트: 성공

## 다음 단계 제안
- 커밋: `/commit`
- PR 생성: `/pull-request`
- 추가 작업: {있다면}
```

---

## 행동 규칙

- 각 Phase 전환 시 반드시 상태를 보고하라.
- Phase 2에서 독립적인 작업은 Agent tool로 병렬 실행하라.
- Phase 3~5 루프는 최대 3회. 초과 시 사용자에게 판단을 요청하라.
- 사용자가 중단을 요청하면 현재 상태를 보고하고 즉시 멈춰라.
- 민감 파일(`config/config.json`의 `sensitiveFilePatterns`)은 절대 수정하지 마라.
- 성공 기준을 달성하지 못한 채 "완료"라고 보고하지 마라.
