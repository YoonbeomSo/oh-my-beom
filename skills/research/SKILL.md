# Research Skill - 조사 파이프라인

한국어로 응답한다.

## 조사 유형 감지

사용자 입력을 분석하여 조사 유형을 결정한다:

| 유형 | 키워드/패턴 | 초점 |
|------|------------|------|
| **architecture** | 구조, 아키텍처, 설계, 의존성, 모듈 | 프로젝트 구조, 레이어, 의존 관계 |
| **performance** | 성능, 느림, 병목, 최적화, 메모리 | 병목 지점, 리소스 사용, 최적화 기회 |
| **bug** | 버그, 오류, 에러, 실패, 깨짐 | 오류 원인, 재현 경로, 관련 코드 |
| **general** | 그 외 | 요청에 맞는 범용 조사 |

유형을 결정하면 알린다:
```
[Research] {유형} 조사를 시작합니다.
```

---

## Phase 1: Explorer (탐색)

Agent tool (`type: explore`)로 프로젝트를 스캔한다.

### 수행 사항

1. **프로젝트 구조 파악**
   - 루트 디렉토리 구조
   - 주요 설정 파일 (build.gradle, package.json, pyproject.toml 등)
   - 소스 코드 루트 위치

2. **관련 파일 식별**
   - 사용자 요청과 관련된 파일/디렉토리 목록
   - 파일 간 의존 관계 (import/require 분석)
   - 설정 파일에서 관련 항목

3. **조사 유형별 추가 탐색**

   | 유형 | 추가 탐색 |
   |------|----------|
   | architecture | 레이어 구조, DI 설정, 모듈 경계 |
   | performance | 데이터베이스 쿼리, 캐시 설정, 외부 API 호출 |
   | bug | 에러 로그 패턴, 예외 처리 경로, 관련 테스트 |
   | general | 요청 맥락에 따라 판단 |

### 출력

탐색 결과를 구조화하여 Phase 2에 전달한다:
```
파일 맵: [관련 파일 목록 + 역할]
의존 관계: [파일 간 참조 관계]
핵심 발견: [탐색 중 발견한 주요 사항]
```

## Phase 2: Researcher (수집)

Agent tool (`type: explore`)로 심층 조사를 수행한다.

### 수행 사항

1. **패턴 수집**
   - Phase 1에서 식별된 파일들의 코드 패턴 분석
   - 반복되는 패턴 식별 (코딩 컨벤션, 설계 패턴 등)
   - 일관성 검사 (패턴이 깨지는 곳)

2. **문서 분석**
   - README, CLAUDE.md, 주석 등 프로젝트 문서 확인
   - 테스트 코드에서 의도와 기대 동작 파악
   - 커밋 히스토리에서 변경 맥락 파악 (필요시)

3. **조사 유형별 심층 분석**

   | 유형 | 심층 분석 |
   |------|----------|
   | architecture | 의존 방향 검증, 순환 의존 탐지, 레이어 위반 |
   | performance | N+1 쿼리, 불필요한 직렬 처리, 캐시 미스 패턴 |
   | bug | 경계 조건, null 처리, 동시성 문제, 상태 불일치 |
   | general | 사용자 요청에 맞춰 유연하게 수집 |

## Phase 3: Analyzer (분석 및 보고)

Phase 1, 2의 결과를 종합하여 구조화된 보고서를 작성한다.

### 보고서 형식

```markdown
# {조사 제목}

조사 유형: {architecture/performance/bug/general}
조사 범위: {대상 프로젝트/디렉토리/파일}

## 현황

현재 상태를 객관적으로 기술한다.

- 구조: ...
- 주요 패턴: ...
- 수치 (해당 시): 파일 수, 의존성 수, 테스트 커버리지 등

## 문제점

발견된 문제를 심각도 순으로 정리한다.

| # | 문제 | 심각도 | 위치 | 영향 |
|---|------|--------|------|------|
| 1 | ... | 높음/중간/낮음 | 파일:라인 | ... |

## 개선 제안

각 문제에 대한 개선 방안을 제시한다.

| # | 제안 | 난이도 | 기대 효과 |
|---|------|--------|----------|
| 1 | ... | 높음/중간/낮음 | ... |

## 실행 방안

개선 제안을 실행 가능한 단계로 정리한다.

### 단기 (즉시 가능)
1. [액션] -> 대상: [파일/모듈]

### 중기 (계획 필요)
1. [액션] -> 선행 조건: [조건]

### 장기 (검토 필요)
1. [액션] -> 리스크: [고려사항]
```

## 파이프라인 연동

조사 결과를 프로젝트에 저장하여 후속 스킬(`/plan`, `/dev`)이 참조할 수 있도록 한다.

1. `${PROJECT_ROOT}/.dev/` 디렉토리가 없으면 생성한다 (`mkdir -p`).
2. 최종 조사 보고서를 `${PROJECT_ROOT}/.dev/research-report.md`에 Write한다.
3. 저장 완료 시 사용자에게 알린다: "조사 보고서가 .dev/research-report.md에 저장되었습니다. /plan 또는 /dev에서 자동으로 참조됩니다."

## 행동 규칙

- 모든 탐색은 읽기 전용이다. 파일을 수정하지 마라.
- 추측하지 마라. 코드에서 확인할 수 없는 사항은 "확인 필요"로 표시하라.
- 보고서에 코드 스니펫을 포함할 때는 파일 경로와 라인 번호를 명시하라.
- Phase 1, 2는 가능하면 Agent tool로 병렬 실행하라. 단, Phase 2는 Phase 1 결과에 의존하므로 Phase 1 완료 후 실행한다.
- 조사 완료 후 다음 액션을 제안하라: `/plan`으로 계획 수립, `/dev`로 구현 시작, 또는 추가 조사.
