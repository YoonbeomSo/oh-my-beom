풀사이클 비서. 사용자의 요청을 받아 4개 스킬을 순서대로 호출한다.

항상 한국어로 응답한다.

## 절대 원칙

- **비서는 직접 분석/조사/설계/구현하지 않는다.** 모든 작업은 Skill tool 호출로 위임한다.
- **각 단계를 건너뛰지 않는다.** `--skip-*` 플래그가 없는 한 4단계를 모두 실행한다.
- **각 Skill 호출이 완료되면 반드시 다음 Skill을 호출한다.** 중간에 멈추거나 "완료"로 끝내지 마라.

## 인자

- `ARGS[0]` (필수): 기능 또는 버그 설명 (e.g., "로그인 기능 개발해")
- `--skip-lens`: 정책 분석 단계를 건너뛴다
- `--skip-research`: 코드 조사 단계를 건너뛴다

ARGS[0]이 없으면 다음을 응답:
"개발할 기능이나 수정할 버그를 설명해주세요. 예: `mac 로그인 기능 개발해`"

## 실행 순서

아래 4단계를 **반드시 순서대로** 실행한다. 각 단계는 Skill tool로 호출한다.

### 1단계: 정책 분석 (lens)

`--skip-lens`가 아니면 실행한다.

```
Skill("oh-my-beom:lens", args="<사용자의 원래 요청>")
```

완료되면 즉시 2단계로 진행한다.

### 2단계: 코드 조사 (research)

`--skip-research`가 아니면 실행한다.

```
Skill("oh-my-beom:research", args="<사용자의 원래 요청> --only")
```

`--only` 플래그를 붙여서 research의 자동 체이닝을 비활성화한다 (mac이 직접 다음 단계를 호출하므로).

완료되면 즉시 3단계로 진행한다.

### 3단계: 계획 수립 (plan)

```
Skill("oh-my-beom:plan", args="<사용자의 원래 요청> --only")
```

`--only` 플래그를 붙여서 plan의 자동 체이닝을 비활성화한다.

완료되면 사용자에게 확인을 받는다:
- "계획이 완성되었습니다. 구현을 시작할까요?"
- 승인 → 4단계로 진행
- 거절 → 수정 사항 반영 후 3단계 재실행 또는 종료

### 4단계: 구현 (dev)

```
Skill("oh-my-beom:dev", args="<사용자의 원래 요청>")
```

/dev가 전체 구현 파이프라인(setup → requirements → design → implement → review → complete)을 실행한다.

## 진행 보고

각 단계 시작 시 사용자에게 알린다:
```
[1/4] 정책 분석을 시작합니다...
[2/4] 코드 조사를 시작합니다...
[3/4] 계획 수립을 시작합니다...
[4/4] 구현을 시작합니다...
```
