# QA 리뷰 출력 포맷

`agents/qa-manager.md`의 페르소나가 코드 리뷰 시 사용하는 출력 템플릿. Codex(Tier C)도 이 파일을 Read하여 동일 포맷으로 응답한다.

## 첫 줄 판정 (필수)

```
## 판정: PASS / FAIL (Critical N건)
```

- **PASS**: Critical 이슈 없음. 커밋 가능.
- **FAIL**: Critical 이슈 있음. 수정 필요. 각 Critical 항목에 파일:라인과 수정 방안을 명시.

## 본문 포맷

```
## 코드 리뷰 결과

### 변경 범위
- 변경된 파일 목록과 각 파일의 변경 요약

### 확실한 문제 (바로 수정)

#### [Critical] 반드시 수정 필요
항목이 없으면 이 섹션 생략.
각 항목:
- **파일명:라인번호** [단계태그] - 문제 설명
  - 현재 코드가 왜 문제인지 구체적으로 설명
  - 수정 방안 제시 (가능하면 코드 예시 포함)

#### [Warning] 수정 권장
항목이 없으면 이 섹션 생략.
각 항목:
- **파일명:라인번호** [단계태그] - 문제 설명
  - 현재 코드의 위험성 설명
  - 개선 방안 제시

#### [Info] 개선 제안
항목이 없으면 이 섹션 생략. 가장 임팩트 있는 3개 이내로 제한한다.
각 항목:
- **파일명:라인번호** [단계태그] - 제안 내용

### 확인이 필요한 사항

1. **파일명:라인번호** - [질문] 맥락 설명
   - 선택지 A: ...
   - 선택지 B: ...
2. ...

없으면: "추가 확인 사항 없음. 리뷰가 완료되었습니다."

**중요**: "추가 확인 사항 없음" 문자열은 Q&A 사이클 종료 신호로 사용된다. 이 문구를 다른 표현으로 바꾸지 말 것.

**단계태그**: 각 발견 항목에 해당 리뷰 단계 태그를 포함한다: `[PERF]`, `[MAINT]`, `[SPEC]`, `[CLEAN]`, `[SPRING]`. 1단계(SCOPE)는 범위 파악이므로 발견 항목에 태그를 붙이지 않는다.

### AC 체크리스트

PRD가 제공된 경우에만 출력한다. 수용 기준 각 항목을 1:1 대조한다.
AC가 명시되지 않은 경우 (--phase review 단독 실행 등) 이 섹션을 생략한다.

- [x] AC-1: {수용 기준 내용} — 충족. {근거 파일:라인}
- [ ] AC-2: {수용 기준 내용} — 미충족. {미충족 사유}
- [x] AC-3: ...

미충족 항목은 CERTAIN > Critical로도 보고한다.

### 탐색 추가 항목
코드 맵에 없는 관련 파일을 발견한 경우 기록한다.
항목이 없으면 이 섹션 생략.
- <파일경로> → 역할 설명

### 총평
- 잘 구현된 점을 1-2개 구체적으로 명시한다 (설계 충실도, 코드 품질, 패턴 활용 등).
- 이어서 전체적인 코드 품질 평가 (1-2 문장).
- Critical 이슈가 있으면 수정을 강력히 권고.
- 문제가 없으면 "리뷰 통과." 로 마무리.
```

## 웹 테스트 필수 마커

코드 리뷰 시 다음 조건 중 하나라도 해당하면 리뷰 결과 마지막에 **반드시** 추가:

```
### [WEB-TEST-REQUIRED]
Playwright 웹 테스트가 필수입니다. 오케스트레이터는 서버를 기동하고 즉시 실행해야 합니다.
탐지 근거: {해당 조건 번호와 파일 경로}
```

**탐지 조건:**
1. **프론트엔드 변경**: 변경 파일에 `*.tsx`, `*.jsx`, `*.vue`, `*.svelte`, `*.html` 포함
2. **어드민/백오피스 변경**: 변경 경로에 `admin`, `backoffice`, `back-office`, `management`, `store-admin` 포함
3. **UI 로직 변경**: 변경 diff에 컴포넌트 렌더링, 라우팅, 폼 로직 변경이 포함된 경우

**중요**: 이 섹션은 반드시 `### [WEB-TEST-REQUIRED]` 라인으로 시작해야 한다(훅 패턴이 라인 시작을 검사).

"요청 범위가 아니다", "코드 리뷰만 요청받았다", "서버가 없다" 등의 이유로 생략 불가.

## 사용자 답변 반영 (Q&A 사이클)

이전 라운드의 질문에 대한 답변을 받으면:
1. 답변을 평가에 반영한다.
2. 답변이 우려를 확인하면 QUESTION에서 CERTAIN으로 승격하고 수정 방안을 제시한다.
3. 답변이 우려를 해소하면 해당 항목을 제거한다.
4. 이전 피드백에 따라 수정된 코드를 재리뷰한다.
5. 새로운 질문이 있으면 포함한다. 없으면: "추가 확인 사항 없음. 리뷰가 완료되었습니다."
